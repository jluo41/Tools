🎯 haipipe-toolkit v2 — Architecture Overview
==============================================

Two-tier skill plugin. Tier 1 user-facing umbrellas parse intent and
dispatch (via Skill()) to Tier 2 specialists. Each umbrella also owns
SHARED ref/ and fn/ docs that specialists read.


─§ 🧑 User types one umbrella command ─────────────────────

+--------------------------------------------------------------------+
|                  🧑 User types one umbrella command                 |
|   /haipipe-data | /haipipe-nn | /haipipe-end                       |
|   /haipipe-project | /haipipe-subject                              |
+--------------------------------------------------------------------+
                                |
                                v
+--------------------------------------------------------------------+
|             ⚙️ Umbrella parses intent on its routing AXIS           |
+--------------------------------------------------------------------+


─§ 📐 Routing axes — one per umbrella ──────────────────────

📐 Routing axes — one per umbrella
-----------------------------------

  +--------------------+----------------------+----------------------+
  | Umbrella           | Axis                 | Specialist names     |
  +--------------------+----------------------+----------------------+
  | /haipipe-data      | STAGE                | -source -record      |
  |                    |                      | -case -aidata        |
  +--------------------+----------------------+----------------------+
  | /haipipe-nn        | LAYER                | -algo -tuner         |
  |                    |                      | -instance -modelset  |
  +--------------------+----------------------+----------------------+
  | /haipipe-end       | 4-AXIS (see below)   | -endpointset         |
  |                    |                      | -meta -trig -post    |
  |                    |                      | -src2input -input2src|
  |                    |                      | -develop-{sagemaker, |
  |                    |                      |  databricks, local}  |
  |                    |                      | -deploy-{sagemaker,  |
  |                    |                      |  databricks, local,  |
  |                    |                      |  mlflow}             |
  +--------------------+----------------------+----------------------+
  | /haipipe-project   | RISK PROFILE         | -new (build)         |
  |                    |                      | -inspect (read)      |
  |                    |                      | -organize (modify)   |
  +--------------------+----------------------+----------------------+
  | /haipipe-subject   | (no decomposition)   | (handled inline)     |
  +--------------------+----------------------+----------------------+

  🚀 /haipipe-end's 4 axes (the only umbrella that's 4-D):
    🧠 Fn-type axis      meta | trig | post | src2input | input2src
                         → /haipipe-end-<fn-type>
    🔧 Artifact axis     package | test | review-overall | dashboard
                         → /haipipe-end-endpointset
    🏗️ Develop axis      develop <target> + verb (test/monitor/teardown/review id)
                         → /haipipe-end-develop-<target>
                         BUILDS the Endpoint_Set
    🚀 Deploy axis       deploy <target> + verb (test/monitor/teardown/review id)
                         → /haipipe-end-deploy-<target>
                         SERVES the Endpoint_Set

  💡 Develop and deploy share targets (sagemaker/databricks/local) but
     operate on different lifecycle phases of the same Endpoint_Set
     artifact: develop BUILDS, deploy SERVES.


─§ 🔀 Three dispatch modes ─────────────────────────────────

🔀 Three dispatch modes
------------------------

  🔀 Mode 1 — Single dispatch (the common case)
     User gives args that resolve to (axis-value, verb).
     Umbrella issues exactly one Skill() call.

         🧑 user
           |
           v
         🎯 umbrella  ──Skill(...)──▶  🔧 specialist
                                            |
           ◀──── status/summary/artifacts/next ────

  🔁 Mode 2 — No-args cross-scope dashboard (parallel fan-out)
     User types just /haipipe-X with no args.
     Umbrella issues N parallel Skill() calls, concatenates the tails.

         🎯 umbrella
           ├──Skill(specialist-1, "dashboard")─▶  tail 1
           ├──Skill(specialist-2, "dashboard")─▶  tail 2
           ├──Skill(specialist-3, "dashboard")─▶  tail 3
           └──Skill(specialist-N, "dashboard")─▶  tail N
                          │
                          v
                  🧑 user gets unified view

  💡 Mode 3 — Inline (umbrella handles itself, no dispatch)
     Cross-scope concerns the umbrella owns directly.

         haipipe-data    "explain" / "L0"  → reads ref/0-overview.md
         haipipe-nn      "L0" / "overview" → reads ref/overview.md
         haipipe-end     no-args dashboard → cross-scope view (artifact + 2 develop +
                                              3 deploy targets; per-Fn-type skipped to
                                              avoid duplication; deferred targets excluded)
         haipipe-end     overview / explain → reads ref/0-overview.md (+ optional fn-type ref)
         haipipe-project no-args overview  → cross-project list
         haipipe-subject all 8 commands    → fully inline


─§ 📋 Specialist return contract (consistent across all umbrellas) ────

📋 Specialist return contract (consistent across all umbrellas)
----------------------------------------------------------------

   +-----------+----------------------------------------------+
   | status    | ok | blocked | failed                        |
   | summary   | 2-3 sentences on what was done               |
   | artifacts | [paths created/read/modified, ids, URLs]     |
   | next      | suggested next command                       |
   +-----------+----------------------------------------------+

   If status != ok → umbrella surfaces summary, stops chaining.

─§ 📌 Discipline ───────────────────────────────────────────

📌 Discipline
--------------

  ✅ Specialists never modify state owned by another specialist.
  ✅ Target deployers (-deploy-*) READ Endpoint_Set produced by -endpointset
     and/or -develop-* ; they never write to it.
  ✅ Target developers (-develop-*) WRITE the Endpoint_Set; they consume
     ModelSet code from /haipipe-nn but never modify it.
  ✅ Each specialist also has its own ref/concepts.md for axis-specific
     content (stage/layer/target/risk-profile).
  ✅ Skill-first development: skill prompt is the source of truth.
