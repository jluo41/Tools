probe TASK-RUNS DASHBOARD — proposed vs existing runs, grouped by arm
=====================================================================

(persisted file: probes/TASK-RUNS.md — the campaign-level EXECUTION view: which
 runs exist vs proposed across all probes. Sits beside STATUS.md. The DISTILLED
 side is NOT a separate dashboard — per-probe DIKW lives in each CYCLE.md, and
 cross-probe DIKW (I + meta-K + W) lives in insights/INDEX.md.)


What this is
------------

The "how many runs, and which are done?" view. A probe is a COMPARISON, and a
single run per arm has no seed-level noise floor — so each arm should plan
MULTIPLE runs (seeds, or config variants) and the dashboard shows, per arm, the
gap between PLANNED and DONE.

Rendered by: /haipipe-probe inspect runs [<probe> | project]
  <probe>  → one probe's arms × runs (persists probes/<…>/RUNS section or inline)
  project  → all probes' arms × runs, project-wide (persists probes/TASK-RUNS.md)

Data model (already in probe-yaml-schema.md):
  arms.<arm>.run_specs[]  = PLANNED runs. Each: {name, seed, status, params}.
                            status ∈ existing | done | proposed | running | failed
  arms.<arm>.runs[]       = ACTUAL linked run-paths on disk (have metrics.json).
  The dashboard = run_specs (plan) JOINED to runs (reality) per arm.

  status meanings:
    existing  a pre-existing run reused as-is (e.g. a fixed baseline model)
    done      planned for THIS probe and already run + linked
    proposed  planned, NOT yet run (the "new runs proposed" the user asks for)
    running   currently executing
    failed    ran but errored / unusable


Per-probe dashboard skeleton
----------------------------

```
RUN DASHBOARD — <P.x0N> <slug>
arm/baseline_arm marked ◆ ; treatment ▶

  ARM           seed  status      run path / (proposed)                  metric
  ─────────────────────────────────────────────────────────────────────────────
◆ <baseline>    42    ✅ existing  tasks/.../run_...                       <V or —>
▶ <treatment>   42    ✅ done      tasks/.../run_...                       <V>
▶ <treatment>    7    ⬜ proposed  (run: bash .../runs/run_..._seed7.sh)   —
▶ <treatment>   13    ⬜ proposed  (run: bash .../runs/run_..._seed13.sh)  —

  FILL:  baseline 1/1 · treatment 1/3   →  seed replication INCOMPLETE
  AGG:   <statistic> — with 1 seed the CI is row-bootstrap only (no seed
         variance); ≥3 seeds enables mean_std_paired_t across seeds.
  TODO:  2 proposed runs → `/haipipe-probe bridge <probe>` materializes them.
```


Project-wide dashboard skeleton (probes/TASK-RUNS.md)
------------------------------------------------

```
# <PROJECT> — Run Dashboard            (derived; regenerate via inspect runs)

| probe | arm | planned | done | proposed | existing | fill | next |
|---|---|---|---|---|---|---|---|
| P.A01 | dr_learner | 3 | 1 | 2 | — | 1/3 | run seed 7,13 |
| P.A01 | slearner_v3 | 1 | — | — | 1 | 1/1 | (baseline reused) |
| P.B01 | k4_bands | 1 | 1 | — | — | 1/1 | (exploratory) |
| ...   | ...        |   |   |   |   |     |    |

Totals: N runs done · M proposed (not yet run) · K existing reused.
Exploratory probes (1 seed/arm) flagged; confirmatory needs ≥3.
```


Render rules
------------

- Derived from probe.yaml run_specs × runs. Regenerate any time; never hand-edit.
- Group rows by arm; mark baseline_arm ◆ and treatment_arm ▶.
- "proposed" rows show the COMMAND that would materialize them (so the dashboard
  doubles as a to-do list). "done"/"existing" rows show the run path + metric.
- Flag exploratory (1 seed/arm, row-bootstrap CI only) vs confirmatory (≥3 seeds,
  seed-level mean±std). A probe is not "strong" until the proposed runs are run.
- Pairs with: probe-status-template (campaign) and probe-cycle-audit (one cycle).
  The run dashboard is the RUN-LEVEL zoom inside a single probe.
