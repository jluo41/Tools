─§ -task: three scopes (the build hierarchy) ───────────────

  /haipipe-project task                ❓ asks which scope
        │
        ├──▶ 📦 project       scaffold a new project
        │       └─▶ examples/Proj{Series}-{Cat}-{Num}-{Name}/
        │            ├── tasks/
        │            ├── diagram/      🖼️ (01-story, 02-boundary, 03-exploration)
        │            └── paper/        📰 optional
        │
        ├──▶ 📁 task-group    scaffold tasks/{G}{NN}_{name}/
        │       ├── sbatch/env.sh     ⚙️ env stub
        │       └── diagram/           🖼️ (01-overview, 02-tasks, 03-progress, 04-design)
        │

─§ Task-type → seed contract (what -task scaffolds) ────────

  🚀 Running process is the SAME for every type — papermill, always.
     Type only decides config seed + where results land.

+----------+----------------+-------------------------+----------------------------+
| 🏷️ Type  | 📁 Group hint  | 🟦 Config seed          | 📊 Results land in         |
+----------+----------------+-------------------------+----------------------------+
| 🧠 (M)   | 🅰️  A-series   | 5_model_<name>.yaml     | _WorkSpace/5-ModelStore    |
| 📊 (E)   | 🅱️  B-series   | eval_<target>.yaml      | results/<run>/metrics.json |
| 🖼️  (F)   | 🆑 C-series    | figure_<name>.yaml      | results/<run>/*.pdf, *.tex |
| 🔧 (D)   | 🆔 D_demo      | {1..4}_<layer>.yaml     | _WorkSpace/{1..4}-*Store   |
| ❔ (X)   | (any)          | (none required)         | results/<run>/             |
+----------+----------------+-------------------------+----------------------------+


─§ Running process — papermill, every task ─────────────────

  📁 runs/run_<variant>.sh
        │
        ▼ Step 1: convert .py → template .ipynb
        🐍 {NN}_{task_name}.py    ──▶  📓 {NN}_{task_name}.ipynb
            (with # %% [parameters] cell)        (template, at task root)
        │
        ▼ Step 2: papermill inject + execute
        📓 template  +  ⚙️ -p key value ...  ──▶  📓 notebooks/<variant>.ipynb
                                                    (executed run record)

  ✅ Every run leaves a reviewable notebook — no separate tee logs.
  ✅ Authoring stays in .py (diff-friendly); execution lives in .ipynb.
  ✅ Parameters injected into the [parameters] cell — uniform across tasks.


─§ Two structural rules (cross-cutting) ────────────────────

  🚧 Rule 1 — Three-level hierarchy
     project  ▶  task-group {G}{NN}_<name>  ▶  task-folder {NN}_<name>
     ❌ NO flat task-folders directly under tasks/
     📜 NO README.md anywhere — doc surface is diagram/

  🤝 Rule 2 — Track A code ⟷ Track B example pairing
     Every new pipeline Fn or ML model stub in code-dev/ or code/hainn/
     auto-generates a paired example task in tasks/D_demo/.


─§ Verb → specialist routing ───────────────────────────────

  🏗️  task · new · scaffold · create · init       ──▶  🏗️ -task
  🔍 review · audit · check · validate · lint    ──▶  🔍 -inspect (review)
  📝 summarize · summary · docs                  ──▶  🔍 -inspect (summarize)
  📜 inventory · list files · files              ──▶  🔍 -inspect (inventory)
  🗺️  overview · show · status                    ──▶  🔍 -inspect (overview)
  🛠️  organize · reorganize · fix structure · move ──▶  🛠️ -organize


─§ Specialist return contract ──────────────────────────────

  📤 Every dispatch returns:

     status:    ✅ ok  |  🚫 blocked  |  ❌ failed
     summary:   2-3 sentences on what was done
     artifacts: [paths created / read / moved]
     next:      💡 suggested next command
