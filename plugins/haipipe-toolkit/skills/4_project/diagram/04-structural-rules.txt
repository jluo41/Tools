🔒 Structural rules enforced by all 3 specialists
═══════════════════════════════════════════════════════════

  Two rules govern haipipe project layout. They live in the
  umbrella's ref/ folder so all specialists read the same source.

─§ Rule 1 — Two-level task hierarchy ─────────────────────

  tasks/  →  group folders  →  task folders  (always two levels)

  examples/{PROJECT_ID}/
  ├── tasks/                           📂 MANDATORY root for all work
  │   ├── README.md                     project-level task overview
  │   │
  │   ├── A_data/                      📁 group: {G}_{group_name}
  │   │   ├── README.md                 group-level overview
  │   │   ├── A1_pull_raw/             📄 task: {G}{N}_{task_name}
  │   │   │   ├── *.py
  │   │   │   ├── config/
  │   │   │   ├── runs/
  │   │   │   ├── results/
  │   │   │   └── README.md
  │   │   └── A2_clean/
  │   │
  │   ├── B_training/
  │   │   ├── B1_train_stats/
  │   │   └── B2_train_nn/
  │   │
  │   └── sbatch/                       (optional, cross-task SLURM)
  │
  ├── paper/                            (optional, manuscript submodule)
  ├── docs/                             (optional, internal notes)
  └── _old/                             (optional, archived)

  🚫 No flat tasks directly under tasks/.
  🚫 No top-level config/ or results/ at project root.

─§ Rule 2 — Code always has a paired example ─────────────

  Track A (code stubs)              Track B (example task)
  +-------------------+             +----------------------+
  | code-dev/         |             | examples/{PROJ}/     |
  |   1-PIPELINE/     |   ⇄ pair    |   tasks/             |
  |   2-MODELING/     | ─────────▶  |     {G}_{group}/     |
  |                   |   (auto-    |       {G}{N}_{task}/ |
  | code/hainn/       |    generate)|         *.py         |
  | code/haifn/       |             |         README.md    |
  +-------------------+             +----------------------+

  🤝 Every new Fn or model stub on the left auto-generates a
      paired task on the right. -inspect review flags any orphan.

─§ Where the rules live ──────────────────────────────────

  📜 haipipe-project/ref/project-structure.md     Rule 1 (Track B)
  📜 haipipe-project/ref/code-structure.md        Rule 2 (Track A pairing)

─§ Naming conventions ────────────────────────────────────

  Project ID:     Proj{Series}-{Category}-{Num}-{Name}
                  e.g.  ProjB-Bench-01-FairGlucose
                        Series   = A misc · B benchmarking
                                   C models · D EHR
                        Name     = CamelCase

  Group folder:   {G}_{group_name}        e.g.  A_data
                  {G}_{S}_{group_name}    e.g.  A_1_data  (sub-ordered variant)

  Task folder:    {G}{N}_{task_name}      e.g.  B1_train_stats

─§ Violation → fix flow ──────────────────────────────────

  👀 -inspect review                            🔧 -organize --fix
   │                                            │
   ├── 🚩 flat task in tasks/        ───▶       flat-tasks  (move into group)
   ├── 🚩 wrong {G}{N} numbering     ───▶       renumber
   └── 🚩 broken paired-example      ───▶       paired
                                                │
                                                ▼
                                          ✅ -inspect review (re-check)
