🎚️ Risk profile — why three specialists, not one
═══════════════════════════════════════════════════════════

  The three specialists are split by **blast radius**, not by
  topic. allowed-tools differ; -inspect can never write, -organize
  can never silently overwrite. The umbrella picks the right one.

─§ Blast-radius matrix ───────────────────────────────────

  +---------------------+---------+----------------------------+----------------------------+
  | Specialist          | Profile | allowed-tools              | Blast radius               |
  +---------------------+---------+----------------------------+----------------------------+
  | 🏗️ -new             | BUILD   | Bash Read Write Edit       | 🟡 creates NEW dirs only   |
  |                     |         | Grep Glob                  |    (refuses overwrite)     |
  +---------------------+---------+----------------------------+----------------------------+
  | 👀 -inspect         | READ    | Read Grep Glob Bash        | 🟢 zero writes             |
  |                     |         | (no Write, no Edit)        |    (summary.md via Bash    |
  |                     |         |                            |     redirect, scoped to    |
  |                     |         |                            |     own project folder)    |
  +---------------------+---------+----------------------------+----------------------------+
  | 🔧 -organize        | MODIFY  | Bash Read Write Edit       | 🔴 git mv / rename / edit  |
  |                     |         | Grep Glob                  |    (highest among the 3)   |
  +---------------------+---------+----------------------------+----------------------------+

─§ Discipline of -organize (the dangerous one) ───────────

  ✅ --dry-run supported and recommended for first pass
  ✅ change plan ALWAYS printed before any operation runs
  ✅ uses `git mv` so changes are tracked & reversible
  🚫 refuses if uncommitted work sits outside target paths
  🛑 on mid-plan error: STOP — report done / failed / remaining

─§ Function axes ─────────────────────────────────────────

  +-----------+-----------------------------------------------+
  | -new      | new (only — purpose-built)                    |
  | -inspect  | overview · review · summarize · inventory     |
  | -organize | organize (only — purpose-built)               |
  +-----------+-----------------------------------------------+

─§ Decision: which specialist for this request? ──────────

           ┌──────────────────────┐
           │ Does it create new   │  yes
           │ files in a fresh     │ ─────▶  🏗️ -new
           │ project_id?          │
           └──────────┬───────────┘
                      │ no
                      ▼
           ┌──────────────────────┐
           │ Does it move /       │  yes
           │ rename existing      │ ─────▶  🔧 -organize
           │ files?               │
           └──────────┬───────────┘
                      │ no
                      ▼
                 👀 -inspect
                 (overview / review /
                  summarize / inventory)

─§ Why segregate at all? ─────────────────────────────────

  🛡️ Separating tools prevents an "audit" request from accidentally
      mutating the repo. The orchestrator never grants Write to
      -inspect, so even a misfire stays read-only.
  🩻 Reviewers can grep allowed-tools to know what each skill can
      possibly touch — no need to read the body.
