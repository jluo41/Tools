# JL comment lifecycle: respond FIRST, remove only after review

## Rule

When JL leaves `> JL:` comments in the outline, the `> CC:` response is the FIRST action CC takes -- before any edit, before any search, before any agent launch. The response IS the work product. Everything else follows from it.

## Workflow (strict order)

1. **Read** all `> JL:` comments in the file
2. **For each comment, IMMEDIATELY write `> CC:` response** underneath it in the outline -- this is step ONE, not a backfill
3. **Then** do whatever the comment requires (search, edit, etc.)
4. **Keep both JL comment and CC response in place** -- never silently remove
5. **Only archive to `_LOG_`** after JL reviews the batch and confirms

## What NOT to do

- Do NOT address the issue first and add the CC response later as an afterthought
- Do NOT silently remove JL comments after resolving them
- Do NOT assume a comment is resolved just because CC took action -- JL must confirm
- Do NOT launch agents or make edits BEFORE writing the CC response

## Why

JL was angry because CC kept doing work (launching agents, editing tex, adding bib entries) without leaving a trail in the outline. The outline is the communication channel. If CC does work but doesn't respond in the outline, JL has no way to track what happened. The CC response is not a formality -- it IS the deliverable that JL reviews.
