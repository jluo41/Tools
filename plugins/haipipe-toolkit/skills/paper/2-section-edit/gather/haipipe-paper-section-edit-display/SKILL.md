Skill: haipipe-paper-section-edit-display
=========================================

display gather worker for `haipipe-paper-section-edit`. Owns the display
lifecycle for one section: identifying what figures/tables the section needs,
planning which display units serve which claims, routing generation to
task-folders, and linking outputs back to the section.

```
/haipipe-paper-section-edit-display                       → status dashboard
/haipipe-paper-section-edit-display audit <section>       → Phase 1: what displays needed?
/haipipe-paper-section-edit-display plan <section>        → Phase 2: map claims to display units
/haipipe-paper-section-edit-display route <section>       → Phase 3: dispatch to task-folders
/haipipe-paper-section-edit-display link <section>        → Phase 5: connect outputs to section
/haipipe-paper-section-edit-display review <section>      → Phase 6: pre-submission display check
```

## Hard Boundaries

1. **NEVER create ad-hoc plots inline.** All figures and tables route through
   display units in `0-displays/` and are generated by task-folders. The paper
   layer plans, it does not plot. "Plot it" = route to a generating task.

2. **NEVER fabricate display content.** Display units contain real data
   outputs. The agent plans layout and routing, not data.

3. **Display units are the working documents.** Each display unit lives in
   `0-displays/displayNN/` with its own `float.tex`, rendered output, and
   source data. The agent links these to the section, not the other way.

4. **JL's display comments go into `0-lifecycle/4-display/4-display.tex`**
   as `%% {JL}: ...` (verbatim, kept across iterations), NOT into per-unit
   `float.tex`.


## Six Phases

```
Phase 1: AUDIT        what displays does this section need?
Phase 2: PLAN         which display unit serves which claim/beat
Phase 3: ROUTE        dispatch generation to /haipipe-task
═══════════════════   HUMAN GATE ═══════════════════════════════════════════
Phase 4: (human)      user reviews generated displays, approves or revises
Phase 5: LINK         connect approved display outputs to section tex
Phase 6: REVIEW       pre-submission display check (content matches claims)
```


## Phase 1: AUDIT

Read the section outline and narrative. For each paragraph, identify:

- Does this paragraph make a claim that needs visual support?
- Is there a `\ref{fig:...}` or `\ref{tab:...}` already in the tex?
- Does the _CITATION_ or _VALUES_ map reference a display?

Output: a gap list showing which paragraphs need displays and which already
have them.

```
P#  | Claim/beat | Display needed? | Current display | Status
P1  | variation  | no              | --              | --
P4  | results    | yes (table)     | display03       | ✅ linked
P5  | gradient   | yes (figure)    | --              | 🔍 needed
```


## Phase 2: PLAN

For each needed display, plan:

- **What type**: figure (plot), table, diagram, illustration
- **What it shows**: which claim/beat it supports
- **Which display unit**: existing `0-displays/displayNN/` or new unit needed
- **Data source**: which task-folder output feeds it

Write the plan to `_DISPLAY_N-section.md` (or update the narrative's
`_DISPLAY_` file if it exists at the paper level).


## Phase 3: ROUTE

For each planned display that needs generation:

1. Check if a task-folder already exists for this display
2. If not, scaffold one via `/haipipe-task` (type = display)
3. If yes, check if the task has run and produced output
4. Dispatch to `/haipipe-task` for execution if needed

The agent does NOT generate the display itself. It routes to the task system.


## Phase 4: HUMAN GATE

The human:
1. Reviews generated display outputs
2. Approves, requests revision, or rejects
3. May add `> JL:` comments on layout, labeling, or content


## Phase 5: LINK

After human approval:

1. Ensure the display unit's `float.tex` is correct
2. Add `\input{0-displays/displayNN/float.tex}` to the section tex if not
   already present
3. Add/update `\ref{fig:...}` or `\ref{tab:...}` in the prose
4. Sync outline with display references
5. Backfill to `0-lifecycle/4-display/` if needed


## Phase 6: REVIEW (pre-submission)

For each display referenced in the section:

1. Does the display content match the claim it supports?
2. Do the numbers in the display match _VALUES_?
3. Is the caption accurate?
4. Does the `\ref` in the prose point to the right display?
5. Is the display rendering clean in the compiled PDF?

## Done criteria

- [ ] All display needs identified
- [ ] All displays planned and mapped to units
- [ ] All display tasks routed and executed
- [ ] All display outputs approved by user
- [ ] All displays linked in section tex
- [ ] _LOG updated with display gather summary


## Anti-patterns

- ❌ Creating ad-hoc matplotlib plots inline ("plot it" = route to task)
- ❌ Editing display content directly instead of re-running the task
- ❌ Placing `\ref{fig:...}` for a display that hasn't been generated yet
- ❌ Putting JL's comments into per-unit float.tex (they go in 4-display.tex)
