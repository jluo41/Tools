probe CYCLE.md — per-probe closed-loop audit
=============================================

What this is
------------

One file per probe that makes a SINGLE cycle fully auditable: click it and see
the whole loop — hypothesis → method (which task ran) → raw evidence → how the
verdict was computed → claim → which insight cards it produced → what's next.

Location: probes/<GROUP>_<slug>/<NN>_<slug>/CYCLE.md
Rendered by: /haipipe-probe inspect cycle <probe>   (read-only; persists the file)
Source of truth stays probe.yaml; CYCLE.md is a DERIVED rollup (regenerate any
time). It does NOT create a hard probe→insight dependency: the forward "insights
filed" list is produced by grepping insights/ for `sources:` containing this
probe id, so deleting an insight just drops a line on next render. Respects the
one-way dependency (insight→probe authoritative; CYCLE.md is a cached view).

Why a derived file and not new fields in probe.yaml: probe.yaml is steering
state the human edits; CYCLE.md is a generated audit surface the human READS.
Keeping them separate means re-rendering never clobbers hand-written intent.


The 7 audit sections (always in this order)
-------------------------------------------

1. HEADER      probe id · title · verdict · one-line claim
2. STAGE TRAIL the 8 stages, each: glyph · artifact path (clickable) · when
               (this is the audit spine — "done" must name the file it produced)
3. HYPOTHESIS  what we asked + resolves_K (the pre-registered question)
4. METHOD      the task that ran: script path, run.sh, key config — so a
               reader can RE-RUN or inspect exactly how the number was made
5. EVIDENCE    metrics.json path + the key numbers (point + CI + N), and the
               aggregation recipe (how delta/CI/status were computed) — so the
               claim is checkable against raw data, not taken on faith
6. VERDICT     result.status + claim + the honest two-level reading
7. PRODUCED    THE PER-PROBE DIKW (the insight, inside the cycle): the 🟦 D this
               probe filed + its own 🟨 K (if it resolved a belief), grepped from
               insights/ `sources:` ; + caveats ; + NEXT (what the cycle implies).

DIKW is PARTITIONED across two levels — this section holds ONLY the per-probe
part. The probe cycle owns 🟦 D + its own 🟨 K (shown here). Cross-probe 🟩 I,
meta-🟨 K, and 🟧 W are NARRATIVE synthesis spanning many probes — they live in
insights/INDEX.md, NOT in any single cycle. So: insight is WITHIN the probe at
the D/K level; the cross-probe rollup is the narrative's insights/ archive.
There is no separate campaign DIKW dashboard (it would just duplicate one or the
other).


Skeleton
--------

```
# CYCLE — <P.x0N> <slug>   [<verdict>]
> <one-line claim>

─§ 1. STAGE TRAIL ───────────────────────────────
  1 design    <glyph>  probes/.../probe.yaml                 <when>
  2 bridge    <glyph>  tasks/<group>/<task>/                 <when>
  3 run       <glyph>  .../results/<run>/metrics.json        <when>
  4 aggregate <glyph>  probe.yaml result:                    <when>
  5 review    <glyph>  (caveats walked)                      <when>
  6 claim     <glyph>  probe.yaml claim:                     <when>
  7 file K/D  <glyph>  insights/.../<cards>                  <when>
  8 explore   <glyph>  (handed to narrative)                 <when>

─§ 2. HYPOTHESIS ────────────────────────────────
  <hypothesis>   resolves_K: <the pre-registered question>

─§ 3. METHOD (re-runnable) ──────────────────────
  script : tasks/<group>/<task>/<build>.py
  run    : bash tasks/<group>/<task>/runs/<run>.sh
  config : <key knobs: model, args, propensity, eval design>

─§ 4. EVIDENCE (checkable) ──────────────────────
  metrics: .../results/<run>/metrics.json
  <metric> = <point> [<ci_lo>, <ci_hi>]  N=<n>  (<ci_method>)
  recipe : <how delta + CI + status were computed; the noise_floor rule used>

─§ 5. VERDICT ───────────────────────────────────
  status : <confirmed|refuted|inconclusive>   (rule that decided it)
  claim  : <claim sentence>
  honest : <two-level reading — what is / isn't established>

─§ 6. PRODUCED (loop closed) ────────────────────
  insights: <D.. / I.. / K.. cards that cite this probe>   (grepped)
  caveats : <bulleted, verbatim from probe.yaml>
  next    : <what this cycle implies for the next probe / narrative>
```


Render rules
------------

- Regenerate from probe.yaml + the linked run's metrics.json + a grep of
  insights/ each call. Never hand-edit (it is derived).
- A stage row names its artifact; an empty path = stage not really done.
- "insights" section: list every card whose `sources:` contains this probe id.
  If none (e.g. inconclusive with no card), say "none (inconclusive → no card)".
- One CYCLE.md per probe. The campaign-wide view stays probes/STATUS.md.
