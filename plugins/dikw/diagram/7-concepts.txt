рҹ—ӮпёҸ  DIKW вҖ” 5 Core Concepts Explained
=====================================

  The five terms form a strict nesting:
  вҡҷпёҸ Stage  >  рҹ”„ Phase  >  вҡЎ Step  >  (рҹ”§ Task | рҹҡҰ Gate)

  Each section below zooms one layer deeper.


в”ҖВ§ рҹҸ—пёҸ  CONCEPT 1 вҖ” STAGE  (outer setup, owned by /dikw) в”Җв”Җв”Җв”Җ

  =========================================================
  рҹҸ—пёҸ  CONCEPT 1 вҖ” STAGE  (outer setup, owned by /dikw)
  =========================================================

  вҡҷпёҸ  Stages are one-time workspace-setup steps that run
  BEFORE the session starts.  They are not analysis.

  +------+  +------+  +-------+  +--------+  +------+  +------+
  | рҹ”ҚS0 |->| рҹ“·S1 |->| рҹҸ—пёҸS2 |->| рҹҸ·пёҸS2.5|->| рҹ‘ҖS3 |->| рҹҡҰS4 |
  |detct |  | snap |  |scaffld|  |gen aim |  |layout|  | gate |
  +------+  +------+  +-------+  +--------+  +------+  +------+
                                                             |
                                                        вң… confirm?
                                                             |
                                           +--------+        v
                                           | рҹ”ҚS4.5 |  +--------+
                                           | explore|  | рҹ”Ғ S5  |
                                           +--------+  |session |  <- рҹ”„ THE LOOP
                                                |      +--------+
                                                |
                                                v  (one-time per рҹ“· snapshot)
                                        рҹ“„ exploration/explore_notes.md

  вҡҷпёҸ  Stages are numbered, finite, and non-repeating.
  рҹ”Ғ  S5 (session) is the only stage that loops internally.


в”ҖВ§ рҹ”„  CONCEPT 2 вҖ” PHASE  (named states inside рҹ”Ғ S5 session) в”Җв”Җв”Җв”Җ

  =========================================================
  рҹ”„  CONCEPT 2 вҖ” PHASE  (named states inside рҹ”Ғ S5 session)
  =========================================================

  рҹ”„  Phases are the states of the analysis state machine.
  The session moves forward through phases.  Any non-forward
  outcome always routes back to рҹ“Ӣ plan вҖ” never cross-phase direct.

       G-рҹ“Ӣ    G-рҹ“Ҡ     G-рҹ“Ҳ     G-рҹ§      G-рҹҺҜ    G-рҹ“қ
         |        |       |       |       |        |
  рҹ“Ӣ в”Җв”Җв”Җ+в”Җв–ә рҹ“Ҡ в”Җв”Җ+в”Җв–ә рҹ“Ҳ в”Җ+в”Җв–ә рҹ§  в”Җ+в”Җв–ә рҹҺҜ в”Җ+в”Җв–ә рҹ“қ в”Җв”Җв–ә вң… done
   в–І          в”Ӯ       в”Ӯ       в”Ӯ       в”Ӯ       в”Ӯ
   в”Ӯ          в”Ӯ       в”Ӯ       в”Ӯ       в”Ӯ       в”Ӯ
   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
          every рҹ”„ "revise [feedback]" routes here
     (рҹ“Ӣ plan re-decides pending_tasks; restarts from plan)

  There are 7 phases:
  рҹ“Ӣ plan | рҹ“Ҡ D | рҹ“Ҳ I | рҹ§  K | рҹҺҜ W | рҹ“қ report | вң… done

  вң… "done" is terminal вҖ” no steps, no gate.


в”ҖВ§ вҡЎ  CONCEPT 3 вҖ” STEP  (sub-structure inside each phase) в”Җв”Җв”Җв”Җ

  =========================================================
  вҡЎ  CONCEPT 3 вҖ” STEP  (sub-structure inside each phase)
  =========================================================

  вҡЎ  Every phase (except вң… done) runs exactly TWO steps, in order.
  No phase skips or re-orders them.

  +------------------------------------------------------------+
  |  рҹ”„ phase = <any of рҹ“Ӣ plan / рҹ“Ҡ D / рҹ“Ҳ I / рҹ§  K / рҹҺҜ W> |
  |                                                            |
  |  вҡҷпёҸ  step 1:  рҹ”§ task   --> execute the work              |
  |       |                                                    |
  |       v                                                    |
  |  рҹҡҰ  step 2:  gate   --> review + decide next move         |
  |       |                                                    |
  |       v                                                    |
  |  outcome: вң… approve / рҹ”„ revise [feedback] / рҹҸҒ done      |
  +------------------------------------------------------------+

  вҡЎ  Step is a POSITION within a phase, not a unit of work.
  "We are at phase=рҹ“ҠD, step=рҹҡҰgate" means D's work is done
  and we are now reviewing it.


в”ҖВ§ рҹ”§  CONCEPT 4 вҖ” TASK  (work items inside step=task) в”Җв”Җв”Җв”Җв”Җ

  =========================================================
  рҹ”§  CONCEPT 4 вҖ” TASK  (work items inside step=task)
  =========================================================

  рҹ”§  A task is a specific named unit of analysis.
  Tasks run one at a time inside step=рҹ”§task.

  +-----------------------------------------------+
  |  вҡҷпёҸ  step = рҹ”§ task                            |
  |-----------------------------------------------|
  |  for each task in pending_tasks[phase]:        |
  |                                               |
  |    рҹ”Қ /dikw-context  <phase> <task>            |
  |    verdict = вң… READY | рҹҡ« BLOCKED | вҸӯпёҸ  SKIP  |
  |                                               |
  |    if вң… READY:                               |
  |      рҹ”§ /dikw-<phase>  <task>                 |
  |      writes:  рҹ“Ӯ insights/<L>/<LNN>-<slug>/   |
  |                 рҹ“„ report.md                  |
  |                 рҹҗҚ analysis.py  (рҹ“ҠD and рҹ“ҲI) |
  |                                               |
  |    if рҹҡ« BLOCKED:  break -> jump to рҹҡҰ gate   |
  |    if вҸӯпёҸ  SKIP:    mark + next task           |
  |                                               |
  +-----------------------------------------------+

  рҹ“Ӣ  Task counts by phase:

  +--------+-------+------------------------------------------+
  | Phase  | Tasks | What each task does                      |
  +--------+-------+------------------------------------------+
  | рҹ“Ӣplan |   1   | write the analysis plan (YAML)           |
  | рҹ“Ҡ D   |  2-3  | рҹҗҚ CODE: profile raw data, measure qual. |
  | рҹ“Ҳ I   |  2-3  | рҹҗҚ CODE: find patterns, correlations     |
  | рҹ§  K   |  1-2  | рҹ’¬ REASON: explain why patterns exist    |
  | рҹҺҜ W   |  1-2  | рҹ’¬ REASON: recommend actions             |
  | рҹ“қrpt  |   1   | synthesize all findings into final doc   |
  +--------+-------+------------------------------------------+

  рҹ“Ӯ  Task folders are named  {L}{NN}-{slug}/
    L  = рҹ“ҠD | рҹ“ҲI | рҹ§ K | рҹҺҜW   (phase letter)
    NN = 01, 02, ...               (next available within that level)
  Example:  рҹ“Ӯ insights/data/D01-col_overview/


в”ҖВ§ рҹҡҰ  CONCEPT 5 вҖ” GATE  (the single step=gate action) в”Җв”Җв”Җв”Җв”Җ

  =========================================================
  рҹҡҰ  CONCEPT 5 вҖ” GATE  (the single step=gate action)
  =========================================================

  рҹҡҰ  A gate is the quality checkpoint at the END of every phase.
  There is exactly one gate per phase; it reads everything and
  proposes ONE outcome.

  +---------------------------------------------------+
  |  рҹҡҰ step = gate  (identifier: G-<phase>)          |
  |---------------------------------------------------|
  |                                                   |
  |  рҹ”Қ /dikw-gate reads ALL of:                      |
  |    рҹ“„ explore_notes.md    (snapshot-level)        |
  |    рҹ“„ plan-raw-v{N}.yaml  (current plan)          |
  |    рҹ“„ all prior gate files  (gates/*.md)          |
  |    рҹ“„ all insights/*/report.md  (every phase)     |
  |                                                   |
  |  рҹҺӯ applies gate_persona (locked; opt-in override |
  |     for one gate only вҖ” does not mutate state)    |
  |                                                   |
  |  proposes exactly ONE outcome:                    |
  |                                                   |
  |    вң… approve             --> next phase forward  |
  |    рҹ”„ revise [feedback]   --> back to рҹ“Ӣ plan     |
  |    рҹҸҒ done                --> jump to рҹ“қ report   |
  |        (only valid at G-рҹ§ K, G-рҹҺҜW, G-рҹ“қreport)   |
  |                                                   |
  |  acceptance ::= per unattended_timeout:           |
  |    рҹ§‘ attended (null)  вҶ’ wait for human reply     |
  |    вҸі timed (N>0)      вҶ’ wait N s, then auto-accept|
  |    рҹӨ– unattended (0)   вҶ’ auto-accept immediately  |
  |  рҹӨ– orchestrator applies the outcome             |
  +---------------------------------------------------+

  рҹ“Ӣ  Gate IDs (one per phase, in order):
    G-рҹ“Ӣplan  ->  G-рҹ“ҠD  ->  G-рҹ“ҲI  ->  G-рҹ§ K  ->  G-рҹҺҜW  ->  G-рҹ“қreport

  рҹ’ҫ  Gate files are written to:
    рҹ“Ӯ sessions/{aim}/gates/{NN}-G-{phase}.md
  where NN is a session-wide sequential counter (00, 01, 02 ...).


в”ҖВ§ рҹ“Ң  SUMMARY вҖ” how they nest в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  =========================================================
  рҹ“Ң  SUMMARY вҖ” how they nest
  =========================================================

  +----------+---------+---------------------------------------------+
  | Concept  | Scope   | Plain definition                            |
  +----------+---------+---------------------------------------------+
  | вҡҷпёҸ Stage | /dikw   | one-time setup step before the session      |
  | рҹ”„ Phase | session | named state in the analysis state machine   |
  | вҡЎ Step  | phase   | рҹ”§task OR рҹҡҰgate вҖ” two per phase, always    |
  | рҹ”§ Task  | step    | one named work item run during step=task    |
  | рҹҡҰ Gate  | step    | quality checkpoint that ends step=gate      |
  +----------+---------+---------------------------------------------+

  Reading from outermost to innermost:
